# house.kg — разбор доски объявлений и схема парсинга

Документ описывает всё, что мы выяснили о сайте **house.kg** (недвижимость Кыргызстана):
как устроен сайт, какие данные он отдаёт, как мы их извлекаем и какие ловушки в нём
зарыты. Составлено по итогам разведки и прогонов на 1000 объявлений.

Все факты ниже **проверены на живых страницах**, а не взяты из предположений.

---

## 1. Что за сайт и что мы берём

house.kg — крупнейшая доска объявлений о недвижимости Кыргызстана. Содержит и
продажу, и аренду, а также объекты в других странах (Россия, Казахстан, ОАЭ и др.).

**Наш охват:** только Кыргызстан, обе сделки (продажа + аренда), все 7 типов
недвижимости, все регионы.

### Сделки

| Сделка | URL-префикс | Значение поля `deal` |
|---|---|---|
| Продажа | `/kupit-*` | `sale` |
| Аренда | `/snyat-*` | `rent` |

### Типы недвижимости (7 штук, одинаковы для обеих сделок)

| `type` | URL (продажа) | URL (аренда) |
|---|---|---|
| `apartment` | `/kupit-kvartiru` | `/snyat-kvartiru` |
| `house` | `/kupit-dom` | `/snyat-dom` |
| `commercial` | `/kupit-kommercheskaia-nedvijimost` | `/snyat-kommercheskaia-nedvijimost` |
| `room` | `/kupit-komnatu` | `/snyat-komnatu` |
| `land` | `/kupit-uchastok` | `/snyat-uchastok` |
| `dacha` | `/kupit-dachu` | `/snyat-dachu` |
| `parking_garage` | `/kupit-parking-garaj` | `/snyat-parking-garaj` |

### Регионы

Фильтр `?region=<id>`. **Кыргызстан — это только id 1–7.** Всё, что 8 и выше —
другие страны, и в датасет попадать не должно.

| id | `region` | Область |
|---|---|---|
| 1 | `chui` | Чуйская область / Бишкек |
| 2 | `issyk_kul` | Иссык-Кульская |
| 3 | `talas` | Таласская |
| 4 | `naryn` | Нарынская |
| 5 | `jalal_abad` | Джалал-Абадская |
| 6 | `osh` | Ошская |
| 7 | `batken` | Баткенская |
| 8+ | — | ❌ Казахстан, Россия, США, ОАЭ, Узбекистан, Таджикистан, Турция… |

> **Почему мы обходим сайт по комбинациям `тип × регион`, а не через `?region=all`.**
> Во-первых, `region=all` тянет другие страны. Во-вторых — и это главное — при обходе
> по комбинации мы **знаем тип и регион каждого объявления заранее**, из самого URL
> потока. Не надо угадывать их из текста. Тип и регион приходят бесплатно и без ошибок.

### Пагинация

`?region=<id>&page=<n>`, ровно **10 объявлений на странице**. Номер последней
страницы берётся из ссылки «Последняя» в блоке `.pagination`.

## 3. Анатомия страниц

### 3.1 Страница списка (`/kupit-kvartiru?region=1&page=2`)

Карточка объявления — `div.listing`. Ссылка на объявление — `p.title a[href]`,
формат `/details/<house_kg_id>`. Из списка мы берём **только URL** — все данные
парсим с детальной страницы (там их кратно больше).

### 3.2 Детальная страница (`/details/<id>`)

| Данные | Селектор | Примечание |
|---|---|---|
| Заголовок | `h1` | «3-комн. кв., 46 м2» |
| Адрес | `.address` | «Бишкек, Магистраль, Аалы Токомбаева 21/4» |
| Цена $ | `.price-dollar` | «$ 198 000» или «$ 1 200/мес.» |
| Цена сом | `.price-som` | «17 360 640 сом» |
| **Координаты** | `#map2gis` → `data-lat`, `data-lon` | карта 2ГИС |
| Описание | `.description-text` | свободный текст автора |
| Характеристики | `.info-row` → `.label` + `.info` | ключ/значение, состав зависит от типа |
| Дата размещения | `.added-span` | «Добавлено 1 день назад» |
| Дата поднятия | `.upped-span` | «Поднято 5 часов назад» |
| **Просмотры** | `.view-count` | число |
| Фото | `a[data-full]` | `..._1200x900.jpg` |
| Автор | `#block-user` | см. §5 |
| Рейтинг/отзывы | `#reviews-modal` | см. §6 |
| ЖК | `.c-name a[href*="jilie-kompleksy"]` | если объект в жилом комплексе |

**Лайков на сайте нет.** Публично отдаётся только счётчик просмотров. То, что
похоже на лайки — кнопки шеринга в соцсети, без счётчика.

## 4. Схема данных (реляционная)

Отзывы и рейтинги **не принадлежат объявлению** — они принадлежат компании или
ЖК, которые шарятся сотнями объявлений. Поэтому данные разложены на 4 таблицы,
связанные стабильными ключами.

```
listings ──┬── author_user_id ──→ users      (частник, автор объявления)
           ├── company_slug   ──→ companies  (агентство/бизнес-аккаунт)
           └── complex_slug   ──→ complexes  (жилой комплекс)

foto/  — все фотографии плоско, имена = uuid4, ссылка из listings.foto_ids[]
```

### Почему именно так

На 1000 объявлений: **600 объявлений ссылаются всего на 76 компаний**. Если хранить
отзывы прямо в объявлении, отзывы одного агентства скопируются десятки раз (на
полном прогоне — тысячи). Вынос в отдельную таблицу:

* убирает дублирование (дедупликация ~8–12×);
* снимает лимит в 20 отзывов — на странице профиля их можно забрать целиком;
* даёт возможность джойнить авторов объявлений с авторами отзывов (единое
  пространство `/user/<hash>`).

### Ключи

| Таблица | PK | Откуда берётся | Стабилен? |
|---|---|---|---|
| `listings` | `id` (наш uuid4) + `house_kg_id` | из URL `/details/<id>` | да |
| `users` | `user_id` | из URL `/user/<hash>` | да |
| `companies` | `slug` | из URL `/<slug>` | да |
| `complexes` | `slug` | из URL `/jilie-kompleksy/<slug>` | да |

### Таблица `users`

Авторы объявлений и авторы отзывов живут в **одном пространстве имён**
`/user/<hash>`, поэтому таблица — их объединение (UNION), а не два разных списка.
Человек, который и объявления размещает, и отзывы пишет, — это одна строка.

| Поле | Описание |
|---|---|
| `user_id` | PK, хэш из URL |
| `name` | имя (часто просто «Пользователь») |
| `ads_count` | сколько объявлений у пользователя |
| `registered_raw` / `registered_date` | «на House.kg с 19 мая 2026» → ISO-дата |
| `is_ad_author` | размещал объявления |
| `is_reviewer` | писал отзывы |

> У компаний пользователя-владельца **нет**: профиль компании не содержит ссылки
> на юзера. Цепочку «компания → её создатель» построить нельзя.

> **Ключевать по slug, а не по названию.** У агентств встречаются почти одинаковые
> имена («Кыргыз Недвижимость» — это три разных slug'а). По названию мы бы склеили
> разные компании и наплодили дубликатов.

## 5. Продавец: «что заявлено» vs «кто по факту»

Это самая тонкая часть сайта — и самая интересная для исследования.

**Заявленный статус** — характеристика `Тип предложения` (`offer_type`):
«от собственника» либо «от агента». Это то, что **написал сам автор**.

**Фактический статус** определяется формой ссылки в блоке автора `#block-user`:

| Ссылка автора | Кто это на самом деле | `seller_type` |
|---|---|---|
| `/user/<hash>` | личный аккаунт | `owner` |
| `/<slug>` (напр. `/megapolis`) | бизнес-аккаунт (тот же slug, что и профиль компании) | `company` |

### Расхождение (замер на 1000 объявлений)

| Заявлено | По факту | Кол-во |
|---|---|---|
| от агента | company | 600 |
| от собственника | owner | 359 |
| **от агента** | **owner** | **41** ← агенты без бизнес-аккаунта |
| от собственника | company | 0 |

Обратного случая (компания, выдающая себя за собственника) в выборке **не нашлось**.
Оба признака хранятся в датасете отдельно, чтобы студенты могли исследовать это сами.

## 6. Рейтинги и отзывы (`#reviews-modal`)

В модалке может быть **два независимых рейтинга сразу**:

| Блок | Чей рейтинг |
|---|---|
| `.modal-body` | **компании/агентства** («рейтинг компании») |
| `.modal-body.alt` | **жилого комплекса** («рейтинг жилого комплекса») |

Их **нельзя смешивать**: плохое агентство ≠ плохой дом.

Состав каждого блока: оценка (`.rating.score`), число отзывов
(`.rating.rate-count`), гистограмма звёзд (`.rating.bars` — 5 строк, сверху 5★),
список отзывов (`.rating.reviews > ul > li`): автор, `user_id`, звёзды, текст, дата.

**Профили сущностей** (откуда берём полные отзывы):
* компания — `house.kg/<slug>`
* ЖК — `house.kg/jilie-kompleksy/<slug>`

## 7. Ловушки сайта (набито на практике)

Каждый пункт — реально сломавший парсер баг, а не теория.

1. **Домен фото плавает.** Один и тот же сайт отдаёт картинки то с
   `cdn.house.kg`, то с `bucket.house.kg`. Фильтровать надо по
   `house.kg/house/images/`, без привязки к поддомену.

2. **У фото нельзя убирать суффикс размера.** Соблазнительно превратить
   `..._1200x900.jpg` в `....jpg`, чтобы получить оригинал — но такой URL отдаёт
   **404**. `data-full` уже указывает на максимальный доступный размер.

3. **Цена аренды не всегда содержит «/мес.»** У части объявлений цена голая
   («$ 2 486» у дома в аренду). Определять период по тексту цены — **ненадёжно**.
   Источник истины — тип сделки, который известен из потока обхода
   (`/snyat-*` vs `/kupit-*`). Суффикс и «Период аренды» лишь различают посуточную
   и помесячную аренду внутри аренды.

4. **Продажа и аренда — несопоставимые цены.** Продажа — полная стоимость
   ($198 000), аренда — ставка ($1 200/мес., $137/сутки). В одной колонке их
   усреднять нельзя, поэтому период вынесен в явное поле `price_period`
   (`total` / `month` / `day`).

5. **Бизнес-аккаунт ≠ наличие контактов.** Определять компанию по ссылке
   `/business/contact/...` **неверно** — она есть только у компаний с заполненными
   публичными контактами. 52 бизнес-аккаунта из 1000 так молча уехали в «частники».
   Правильный признак — форма ссылки автора (§5).

6. **Один смысл — два разных ярлыка.** Аренда пишет «Кол-во комнат», продажа —
   «Количество комнат». Без маппинга в общий ключ в датасете появятся две колонки
   об одном и том же.

7. **Даты относительные и «протухают».** Сайт пишет «2 месяца назад». Такое
   значение бессмысленно вне момента парсинга, поэтому храним и оригинал
   (`posted_raw`), и вычисленную абсолютную дату (`posted_date`).

8. **Отзывы на странице объявления обрезаны до 20**, даже если их 38. Кнопки
   «показать ещё» нет. Полный список — только на профиле компании/ЖК.

9. **Число комнат почти всегда в заголовке**, а не в характеристиках (отдельное
   поле заполнено лишь у ~3%). Достаётся регуляркой из `title`
   («3-комн. кв., 46 м2»).

10. **`requests.Session` не потокобезопасна.** При многопоточном обходе каждому
    потоку нужна своя сессия (`threading.local`).

## 9. Как это парсится

* **BeautifulSoup + lxml**, обычный HTML — JS-рендеринг не нужен, всё в разметке.
* **Многопоточность:** задача чисто сетевая (I/O-bound), поэтому потоки, а не
  процессы — GIL освобождается на ожидании сокета. 10 воркеров.
* **Скорость (замер):** ~11 900 объявлений/час вместе с фото. Полный прогон
  (~25 800 объявлений, ~230k фото, ~46 ГБ) — **порядка 2.5–3 часов**.
* **Вежливость:** 10 потоков сайт держит без единого 429. Есть ретраи с
  экспоненциальной паузой на 429/503.
* **Порядок обхода:** сначала объявления (копим множество slug'ов сущностей),
  затем отдельным проходом — уникальные компании и ЖК. Это сотни страниц вместо
  десятков тысяч.
